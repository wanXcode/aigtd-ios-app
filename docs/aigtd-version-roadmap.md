# AIGTD 版本迭代路线

## 文档目的

本文档用于规划 AIGTD 从当前的单轮任务指令执行器，逐步升级为能够理解上下文、调用工具、执行多步操作、处理歧义并协助用户规划事务的智能助手。

每个版本都应能够独立发布到 TestFlight，并在通过对应验收标准后再进入下一阶段。

## 当前版本基线

当前正式营销版本为 `0.2.0`。

目前版本配置存在不一致：

- Xcode 工程中的版本为 `0.2.0 (3)`
- `project.yml` 中的版本为 `0.2.0 (2)`
- 已发布的 TestFlight 构建为 `0.2.0 (4)`

下一个 TestFlight 版本应从 `0.3.0 (5)` 开始，并同步更新 `project.yml` 和 Xcode 工程，避免重新生成工程后构建号回退。

## 总体产品目标

AIGTD 最终需要具备以下能力：

- 一句话可以包含零个、一个或多个任务操作
- AI 可以先查询 Reminders，再决定如何处理
- 能理解“刚才那条”“第二个”“就按这个来”等上下文指代
- 简单且低风险的操作可以直接执行
- 模糊、批量或高风险操作先展示理解或计划并等待确认
- 执行失败后能够读取真实错误、重新查询并继续处理
- 能逐步记住用户的时间、清单和排期偏好
- 本地规则负责权限、安全和数据校验，不承担主要意图判断

目标运行链路：

```text
用户目标
  -> 获取相关上下文
  -> AI 判断是否需要查询工具
  -> 查询 Reminders
  -> AI 生成回复或操作计划
  -> 策略层判断是否需要确认
  -> 执行一个或多个操作
  -> 检查真实执行结果
  -> 回复用户并更新记忆
```

## 版本路线总览

| 版本 | 版本主题 | 核心目标 |
| --- | --- | --- |
| `0.3.0` | 稳定基线 | 固化当前能力，建立测试、评测和可观测基础 |
| `0.4.0` | 上下文与记忆 | 让 AI 真正理解任务数据和连续对话 |
| `0.5.0` | 工具调用与多动作 | 支持查询后推理以及一次执行多个操作 |
| `0.6.0` | 自然交互与安全 | 支持澄清、文字确认、失败恢复和撤销 |
| `0.7.0` | 智能规划 | 根据任务约束和用户偏好制定事务计划 |
| `1.0.0` | 正式可用 | 完成可靠性、隐私和产品体验收口 |

## `0.3.0`：稳定基线版

详细需求文档：[`v0.3.0-prd.md`](releases/v0.3.0-prd.md)

### 版本目标

把现有功能整理成可以持续迭代的稳定基础，并让后续智能化修改具备可验证、可回归和可定位的条件。

### 主要内容

- 完成并提交当前“先规划、再确认、再重排”的任务时间调整功能
- 统一 Xcode 工程、`project.yml`、发布记录中的版本号和构建号
- 建立至少 100 条中文真实对话评测样例
- 覆盖创建、查看、修改、移动、完成、删除、重排和普通对话
- 增加模型输入、结构化输出、本地操作和最终执行结果的分层调试记录
- 给日期解析、任务匹配和排期逻辑增加单元测试
- 对调试记录中的用户任务和模型凭证进行脱敏
- 修复批量操作部分执行、错误回执和界面状态不一致问题

### 发布验收

- 当前创建、查看、移动、完成、删除和重排功能无明显回退
- 典型对话能够自动回放并给出明确的通过或失败结果
- 错误可以定位到模型理解、参数生成、任务匹配或 EventKit 执行阶段
- 批量操作不会在部分失败后错误宣称全部成功
- 项目版本统一为 `0.3.0`，首个 TestFlight 构建建议为 `0.3.0 (5)`

## `0.4.0`：上下文与记忆版

### 版本目标

让 AIGTD 真正知道用户当前有哪些任务、刚才讨论了什么，以及用户已经确认过哪些偏好。

### 主要内容

- 新增结构化的 `AgentContextSnapshot`
- 按需向模型提供任务 ID、标题、清单、时间、备注和完成状态
- 在需要读取任务数据的请求前刷新 Reminders
- 正式接通 `Prompt`、`Solu`、`Memory` 和 `Operating Guide`
- 在最近对话窗口之外维护会话摘要
- 保存最近创建、修改、展示和选择过的任务 ID
- 将“刚才那条”“第二个”“上一条”等指代解析为稳定任务 ID
- 长期记忆只记录明确且稳定的用户偏好，不自动保存普通聊天内容
- 为发送到远端模型的任务信息提供隐私范围说明和可控选项

### 发布验收

- 连续多轮对话后仍能保持当前任务和讨论目标
- “删除刚才创建的任务”不再依赖标题模糊匹配
- “看看明天的任务，然后把第二条改到下午”可以正确完成
- 同名任务不会被静默错误操作
- Agent 页面修改 Prompt 后，下一轮对话行为能够实际发生变化

## `0.5.0`：工具调用与多动作版

### 版本目标

将 AIGTD 从单 Action 解析器升级为能够查询、思考并连续调用多个本地工具的任务 Agent。

### 核心组件

- `AgentOrchestrator`：管理模型、工具、执行结果和后续推理循环
- `AgentContextBuilder`：按需构建任务和会话上下文
- `ReminderQueryService`：提供稳定、可筛选的任务查询能力
- `AgentTool`：定义强类型的读取和写入工具协议
- `ExecutionResult`：向模型和 UI 返回统一的真实执行结果

### 本地工具

- `search_reminders`
- `get_reminder_details`
- `create_reminder`
- `update_reminder`
- `move_reminder`
- `complete_reminder`
- `delete_reminder`
- `propose_schedule`
- `apply_schedule`

### 主要内容

- 允许模型先查询 Reminders，再根据查询结果继续推理
- 将单个 `action` 协议升级为一个或多个强类型 `operations`
- 支持一句话中的多个连续动作
- 所有修改操作优先使用稳定任务 ID
- 将本地关键词规则缩减为离线兜底、安全校验和参数合法性检查
- 防止远端模型的合法结构化结果被本地规则覆盖
- 限制单轮最大工具调用次数，避免循环失控
- 对工具调用增加超时、错误分类和可取消能力

### 目标协议示例

```json
{
  "goal": "调整会议并增加准备提醒",
  "reply": "我来调整一下。",
  "operations": [
    {
      "tool": "update_reminder",
      "reminder_id": "stable-id",
      "changes": {
        "due_date": "2026-07-13T15:00:00+08:00"
      }
    },
    {
      "tool": "create_reminder",
      "input": {
        "title": "带会议资料",
        "due_date": "2026-07-13T14:30:00+08:00"
      }
    }
  ]
}
```

### 发布验收

- “把会议改到三点，再提醒我带资料”可以完成两个操作
- “把这三条移到工作清单并安排到本周”可以形成完整操作计划
- AI 可以先搜索任务，再决定具体操作
- 工具执行结果能够回到模型，模型可以据此生成最终回复
- 远端 AI 的正确结果不会再被本地关键词规则覆盖

## `0.6.0`：自然交互与安全版

### 版本目标

解决对话生硬、误执行、歧义处理不足和失败后无法继续的问题。

### 主要内容

- 接通现有 `ExecutionPolicy`
- 根据置信度、影响范围和操作风险决定是否确认
- 低置信度时先展示“我理解的是……”
- 同名或近似任务展示候选项供用户选择
- 支持“就按这个来”“执行吧”“第二条不要改”等文字确认和方案调整
- 删除、批量修改和新建清单按执行策略确认
- 为批量操作记录执行前快照
- 支持撤销创建、移动、改时间和完成等操作
- 批量操作显示完整成功、部分成功或失败状态
- 执行失败后允许 AI 读取错误并尝试一次合理修正
- 重新设计 Action 卡片，使其表达用户计划和真实结果，而不是内部日志

### 交互规则

- 普通对话和只读查询不显示执行卡片
- 简单、明确、低风险的单动作可以直接执行
- 多动作和批量操作先展示计划
- 删除等高风险操作按策略确认
- 执行结果必须来自真实的 EventKit 返回值
- 不向用户暴露内部 intent、schema 和调试字段

### 发布验收

- 低置信度判断不会直接触发高风险修改
- 模糊目标会先澄清，不会猜测后静默执行
- 用户可以自然修改上一份方案
- “就按这个来”能够应用当前待确认方案
- 批量操作失败时可以重试、跳过失败项或撤销已完成部分

## `0.7.0`：智能规划版

### 版本目标

让 AIGTD 不仅能操作 Reminders，还能根据任务约束和用户习惯协助安排事务。

### 规划输入

- 截止时间
- 优先级
- 预计时长
- 用户工作时间
- 周末偏好
- 清单和项目关系
- 当前任务密度
- 固定时间与可移动时间
- 用户过去接受或调整过的方案

### 主要内容

- 由 AI 生成具体排期方案，不再以本地平均分散公式作为主要规划方式
- 本地校验日期冲突、过去时间、任务范围和执行权限
- 支持“每天最多三件”“下午不要安排重要任务”等规划约束
- 从用户确认和修改中提取稳定偏好
- 增加每日计划、晚间回顾和每周回顾
- 主动发现逾期任务、长期停滞项目和等待中事项
- 主动建议只展示方案，未经用户允许不修改系统数据

### 发布验收

- 相同任务在不同截止时间、优先级和时间约束下会产生不同排期
- “未来两周依次安排”能够生成有依据的具体方案
- 用户说“周末不要安排”后，可以重新规划当前方案
- 后续方案能够持续遵守已经确认的用户偏好
- AI 可以简短说明规划依据，但不输出内部推理过程

## `1.0.0`：正式可用版

### 版本目标

完成可靠性、隐私、安全和用户体验收口，不在该版本继续引入未经验证的大型能力。

### 发布条件

- 核心对话评测达到稳定目标
- 单动作和多动作执行链路可靠
- 没有已知的高风险误执行问题
- 上下文、记忆、规划、确认和撤销链路完整
- TestFlight 用户连续使用后没有阻塞性问题
- 隐私说明、数据发送范围和凭证管理完成检查
- 崩溃、性能、无障碍和关键设备兼容性完成验证

## 版本与构建号规则

- 新能力使用次版本号，例如 `0.3.0`、`0.4.0`
- 同一能力版本的修复使用补丁版本号，例如 `0.3.1`、`0.3.2`
- 每次上传 TestFlight 都递增构建号
- `project.yml` 是版本配置的主要来源，生成 Xcode 工程后必须核对版本一致性
- 发布说明、`README.md`、`CHANGELOG.md` 和 Xcode 工程版本应保持同步
- 已上传过的构建号不得重复使用或回退

## 质量指标

每个 TestFlight 版本应持续记录：

- 单动作意图识别正确率
- 多动作完整执行率
- 上下文指代正确率
- 错误自动执行次数
- 用户重复表达同一目标的比例
- 工具调用成功率
- 批量操作部分失败率
- 用户修改 AI 方案的比例
- 从发送到首段可见回复的延迟
- 从发送到最终执行完成的延迟

初期先建立真实基线，再逐版本设定目标。最重要的两个产品指标是误执行率和用户重复表达率，这两个指标最能反映 AIGTD 是否真正变得更聪明、更自然。

## 推荐实施顺序

```text
0.3.0 稳定基础
  -> 0.4.0 上下文与记忆
  -> 0.5.0 工具编排与多动作
  -> 0.6.0 自然交互与安全
  -> 0.7.0 智能规划
  -> 1.0.0 正式发布
```

其中 `0.4.0` 和 `0.5.0` 是智能化能力提升最大的两个版本，应优先保证架构质量，不以继续增加本地关键词规则作为主要交付方式。
