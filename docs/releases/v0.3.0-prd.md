## 2. 版本定位

`0.3.0` 是 AIGTD 智能化改造开始前的稳定基线版本。

这一版不重写 Agent 架构，不引入多工具循环，也不解决所有长期上下文问题。它的任务是把当前已经存在的单 Action 链路做稳、做真、做得可测试，为 `0.4.0` 的上下文与记忆改造和 `0.5.0` 的工具编排改造建立可靠基础。

本版本聚焦四件事：

1. 完成“先规划、再确认、再重排”的时间调整能力
2. 保证聊天回执与 Apple Reminders 的真实执行结果一致
3. 建立可以持续回归的自动测试和对话评测基线
4. 建立可定位问题且不泄露敏感信息的诊断链路

## 3. 背景与问题

当前版本已经具备创建、查看、移动、完成、删除和重排提醒事项的基础能力，但仍存在以下稳定性问题：

- 项目版本号和构建号在 `project.yml`、Xcode 工程和 TestFlight 之间不一致
- 当前没有单元测试 target，修改规则或 Prompt 后无法自动确认是否回退
- 远端模型、本地 fallback、Action 解析和 EventKit 执行之间缺少统一的诊断链路
- 当前调试存储可能保存完整远端响应，不具备明确的脱敏规则
- 批量重排逐条执行，中间失败时可能已经修改部分任务
- 批量操作失败时，聊天回复不能准确表达成功项和失败项
- 规划卡片、执行状态和 Reminders 真实数据之间缺少完整一致性约束
- 现有回归主要依赖人工真机测试，典型中文表达没有形成稳定样例集

如果直接在此基础上开发更复杂的上下文和多动作能力，问题会继续叠加，并且很难判断错误发生在哪个阶段。

## 4. 版本目标

### 4.1 产品目标

- 用户表达重排时间需求后，先看到明确方案，不立即修改系统任务
- 用户确认后，系统严格按照已展示的同一份方案执行
- 所有写操作只在 EventKit 成功后显示成功回执
- 批量操作部分失败时，用户能知道哪些成功、哪些失败
- 当前单 Action 能力在升级后不出现明显回退

### 4.2 工程目标

- 建立自动化单元测试 target
- 建立至少 100 条中文对话评测数据
- 建立统一的 Agent 执行阶段诊断记录
- 明确调试数据的脱敏、保存和清理规则
- 统一版本号和构建号来源
- 形成可重复执行的发布前验收流程

### 4.3 成功标准

- 自动测试在 Debug 和 Release 配置下通过
- 100 条对话评测可以自动运行并输出分类结果
- 核心行为评测通过率达到发布门槛
- 没有“实际执行失败但回复成功”的已知场景
- 批量重排不会把部分成功描述成全部成功
- TestFlight 构建版本与项目文档保持一致

## 5. 非目标

以下能力明确不属于 `0.3.0`，不得为了顺手优化而扩大本版本范围：

- 不实现完整的 `AgentContextSnapshot`
- 不实现长期会话摘要或自动记忆学习
- 不将协议升级为多 `operations`
- 不实现多轮工具调用和 `AgentOrchestrator`
- 不实现 AI 主导的智能排期算法
- 不支持预计时长、优先级、工作时间和任务依赖规划
- 不实现“就按这个来”的纯文字方案确认
- 不实现通用撤销系统
- 不大规模重做 Chat 或 Reminders 页面视觉设计
- 不删除本地 fallback

这些能力分别留给 `0.4.0`、`0.5.0`、`0.6.0` 和 `0.7.0`。

## 6. 目标用户场景

### 6.1 创建任务

用户输入：

```text
明天上午九点提醒我整理数据库
```

期望：

- 创建一条真实 Apple Reminder
- 执行前不宣称成功
- EventKit 保存成功后才回复“已经记好了”
- 结果卡展示任务、时间和清单

### 6.2 查看任务

用户输入：

```text
明天有什么事
```

期望：

- 不修改 Reminders
- 不显示执行结果卡
- 回复与当前同步到的 Reminders 数据一致

### 6.3 修改任务所属清单

用户输入：

```text
把整理数据库移到工作清单
```

期望：

- 唯一匹配时执行移动
- 找不到或存在多个候选时明确失败，不随意选择
- 真实移动成功后再显示成功回执

### 6.4 重排任务时间

用户输入：

```text
把当前过期任务安排到未来两周内
```

期望：

- 生成一份包含任务和新时间的方案
- 规划阶段不修改 Reminders
- 卡片显示“待确认”和“应用这个方案”
- 用户点击后严格执行卡片中保存的方案
- 执行完成后刷新 Reminders 并显示真实结果

### 6.5 批量操作部分失败

场景：

- 方案包含 5 条任务
- 执行前两条后，其中一条任务被系统或其他设备删除

期望：

- 继续或停止策略必须固定且可预测
- 最终结果明确展示成功数量和失败数量
- 不回复“5 条已经全部重排”
- 失败项保留可诊断的非敏感错误信息

## 7. 功能需求

## FR-001 版本号统一

### 需求

- 将营销版本统一为 `0.3.0`
- 首个 TestFlight 构建号使用 `5`
- `project.yml` 作为工程版本配置的主要来源
- 重新生成 Xcode 工程后，版本号不得发生回退
- 同步更新 `README.md`、`CHANGELOG.md` 和 `docs/releases/`

### 验收

- Debug 和 Release 的 `MARKETING_VERSION` 都为 `0.3.0`
- Debug 和 Release 的 `CURRENT_PROJECT_VERSION` 都为 `5`
- Archive 中显示 `0.3.0 (5)`
- `xcodegen generate` 后版本保持一致

## FR-100 重排方案生成

### 需求

- `plan_reschedule` 必须处于 `plan` 或 `apply` 两个明确阶段之一
- `plan` 阶段只生成和持久化方案，不修改 EventKit
- 方案必须保存稳定任务 ID、任务标题、原清单和目标时间
- 同一张卡片展示的数据和最终执行的数据必须来自同一份 `plan_json`
- 方案生成失败时不得进入可应用状态
- 没有可重排任务时，应自然回复并且不展示空方案卡

### 当前规划规则

`0.3.0` 继续使用确定性的本地规划器：

- 默认只处理未完成任务
- 默认从次日上午 09:00 开始
- 默认时间窗为 14 天
- 按当前任务排序在时间窗内均匀分配
- 指定清单时只处理该清单任务
- 指定“过期任务”时只处理到期时间早于当前时间的未完成任务

本版本不宣称该算法是智能排期。UI 文案使用“我先排了一个方案”，不解释为基于优先级或工作量的智能规划。

### 验收

- `plan` 阶段 EventKit 数据完全不变
- 卡片中的每条任务都有稳定 ID 和可解析的 ISO8601 时间
- 相同输入和相同任务快照生成相同方案
- 方案不会包含已完成任务
- 方案中的时间不会早于当前时间

## FR-110 重排方案展示

### 需求

- 规划态标题为“排了个方案”
- 规划态状态为“待确认”
- 主按钮为“应用这个方案”
- 展示范围、任务数量和任务时间预览
- 超过展示上限时明确提示还有多少条任务
- 规划态不得出现“已修改”“已完成”等成功表达

### 状态定义

| 阶段 | 状态 | 主文案 | 主按钮 |
| --- | --- | --- | --- |
| `plan` | ready | 排了个方案 | 应用这个方案 |
| `apply` | pending | 正在重排 | 不可重复点击 |
| `apply` | success | 排好了 | 去看清单 |
| `apply` | partial | 部分排好了 | 查看结果 |
| `apply` | failed | 没排成 | 继续编辑 |

### 验收

- 用户连续点击主按钮不会触发重复执行
- 状态更新过程中卡片不闪回规划态
- 最终卡片状态与实际执行结果一致

## FR-120 重排方案执行

### 需求

- 应用方案时必须使用 `plan_json` 中保存的稳定任务 ID
- 不允许在执行时重新按标题搜索并生成另一份任务集合
- 每条任务记录独立执行结果
- 执行结束后统一刷新 App 内 Reminders 数据
- 执行期间取消重复提交
- 方案执行完成后不可再次应用同一份方案

### 批量执行策略

`0.3.0` 采用“尽力执行并完整汇报”策略：

- 单条失败不阻止其余任务继续执行
- 保存每条任务的成功或失败状态
- 全部成功时状态为 `success`
- 部分成功时状态为 `partial`
- 全部失败时状态为 `failed`
- 最终回复包含总数、成功数和失败数

### 结果示例

全部成功：

```text
已经按方案重排了 5 条任务。
```

部分成功：

```text
这次重排成功 4 条，另有 1 条没有改成。失败的是“整理数据库”，它可能已经被删除或修改。
```

全部失败：

```text
这次方案没有应用成功。5 条任务都没有改动，你可以稍后重试。
```

### 验收

- 单条失败后其余任务仍按方案尝试执行
- `partial` 状态不会显示成 `success`
- 失败任务标题最多展示 3 条，更多内容用数量收口
- 刷新失败和写入失败需要分别记录

## FR-200 Action 回执真实性

### 需求

- 创建、建清单、移动、完成、删除和应用重排都必须以真实本地执行结果为准
- AI 返回的 `reply` 只能作为执行前文案或只读回复
- 写操作执行前不得出现“已经完成”类表达
- EventKit 抛错后必须显示失败结果
- `ActionLog.executionStatus` 必须支持并统一使用：
  - `pending`
  - `success`
  - `partial`
  - `failed`
- 聊天气泡、结果卡和 `ActionLog` 三者状态保持一致

### 验收

- 断开 Reminders 权限后，任何写操作都不会显示成功
- 找不到目标任务或清单时，UI 显示明确失败
- App 切换前后台后，已完成 Action 不重新执行

## FR-210 Action 重复执行保护

### 需求

- 每个写操作使用唯一 `ActionLog.id` 作为本地幂等键
- `pending` 状态可以继续执行
- `success` 和 `partial` 状态默认不可重新执行
- 卡片按钮在执行期间禁用
- View 重建、消息列表刷新和 App 前后台切换不得重复触发执行

### 验收

- 快速连续点击不会创建重复任务或重复修改时间
- 重启 App 后不会自动重放已成功 Action

## FR-300 分阶段诊断记录

### 需求

为每次用户请求生成一个 `traceID`，至少记录以下阶段：

1. `input_received`
2. `local_preview_completed`
3. `remote_request_started`
4. `remote_response_received`
5. `structured_parse_completed`
6. `fallback_resolution_completed`
7. `action_execution_started`
8. `action_execution_completed`
9. `reminders_refresh_completed`
10. `reply_finalized`

每个阶段记录：

- `traceID`
- 时间戳
- 阶段名称
- 成功、失败或跳过状态
- 耗时
- action 类型
- 经过脱敏的摘要
- 错误类别和用户可见错误摘要

### 诊断目标

出现问题时，可以回答：

- 模型是否成功返回
- 返回是否为合法结构化 Action
- 是否启用了本地 fallback
- 最终执行的是哪个 Action
- EventKit 是否成功
- Reminders 刷新是否成功
- 用户最终看到了什么状态

### 验收

- 任意一条 Action 都可以通过 `traceID` 还原执行阶段
- 诊断记录不会因为远端请求失败而中断
- 诊断功能失败不得影响正常任务执行

## FR-310 调试信息脱敏

### 禁止保存

- API Key
- Authorization Header
- 语音 Access Key
- 完整请求 Header
- 用户模型凭证

### 默认脱敏

- 任务标题
- 任务备注
- 用户原始聊天内容
- 模型原始完整响应

默认只保存：

- 文本长度
- 内容哈希或截断后的脱敏摘要
- action 类型
- 字段是否存在
- HTTP 状态码
- 错误类型

开发者显式开启“保存完整调试内容”后，可以在本机短期保存原始文本，但必须满足：

- 开关默认关闭
- 明确提示其中可能包含私人任务内容
- 只保存在本机
- 提供立即清除按钮
- 最多保留最近 20 条或 7 天，先到者触发清理
- 凭证即使在该模式下也必须继续脱敏

### 验收

- 搜索本地诊断导出内容时找不到 API Key
- 默认模式下无法还原完整任务备注和聊天原文
- 用户可以一键清除全部诊断数据

## FR-400 中文对话评测集

### 数据规模

建立至少 100 条固定评测样例，建议分布：

| 类别 | 数量 |
| --- | ---: |
| 创建任务 | 15 |
| 查看任务 | 15 |
| 移动任务 | 10 |
| 完成任务 | 10 |
| 删除任务 | 10 |
| 重排任务 | 15 |
| 普通聊天与试探 | 10 |
| 最近上下文指代 | 10 |
| 错误和异常返回 | 5 |
| 合计 | 100 |

### 样例字段

每条样例至少包含：

```json
{
  "id": "create-001",
  "input": "明天上午九点提醒我整理数据库",
  "fixture": "default-reminders",
  "expected_intent": "create_reminder",
  "expected_entities": {
    "title_contains": "整理数据库",
    "has_due_date": true
  },
  "should_execute": true,
  "should_show_card": true,
  "risk": "low"
}
```

### 执行方式

- 本地规则评测必须离线、确定性运行
- 结构化协议解析评测使用固定模型响应 fixture
- 远端模型评测单独执行，不作为普通单元测试的网络依赖
- 评测结果输出总通过率和各分类通过率
- 失败结果显示样例 ID、期望和实际结果

### 发布门槛

- 创建、查看、完成和普通聊天分类通过率不低于 90%
- 删除和批量重排不得出现高风险误执行
- 相比 `0.2.0` 基线不得出现分类通过率显著下降
- 远端模型评测失败不阻塞离线单元测试，但必须在 TestFlight 发布前人工检查

## 8. 用户交互状态

所有写操作统一遵循以下状态流：

```text
用户输入
  -> interpreting
  -> pending（需要执行时）
  -> success / partial / failed
```

重排操作额外包含规划态：

```text
用户要求重排
  -> plan ready
  -> 用户确认
  -> apply pending
  -> success / partial / failed
```

状态要求：

- `interpreting` 可以展示流式文字，但不得泄露结构化 JSON
- `pending` 必须尽快出现，且不能提前宣称成功
- `success` 必须有真实本地执行依据
- `partial` 必须说明成功和失败数量
- `failed` 必须提供简短、可行动的原因

## 9. 错误分类

错误至少分为以下类别：

| 类别 | 示例 | 用户侧处理 |
| --- | --- | --- |
| configuration | 模型配置不完整 | 引导检查 Agent 设置 |
| network | 超时、连接失败 | 允许重试或使用本地 fallback |
| remote_format | 模型返回非法结构 | 使用 fallback 或自然回复 |
| action_validation | 缺少任务标题或目标清单 | 明确说明无法解析 |
| permission | Reminders 权限关闭 | 引导打开系统权限 |
| not_found | 任务或清单不存在 | 告知目标未找到 |
| ambiguous | 同名候选超过一个 | 展示候选或要求更具体 |
| eventkit_write | EventKit 保存失败 | 告知未成功并允许重试 |
| refresh | 写入后刷新失败 | 告知操作可能已完成，建议刷新确认 |
| partial_execution | 批量操作部分失败 | 展示成功与失败数量 |

内部错误类型可以详细，但用户文案必须简短，不直接展示堆栈、原始 JSON 或内部 schema。

## 10. 数据与隐私要求

- 用户任务仍以 Apple Reminders 为唯一任务事实来源
- `plan_json` 只保存执行方案需要的字段
- 不在日志中保存模型或语音凭证
- 不在默认诊断模式中保存完整用户聊天原文和任务备注
- 调试数据仅保存在本机，不自动上传
- 如果未来需要导出诊断数据，必须由用户主动触发
- 测试 fixture 使用虚构任务，不复制用户真实数据

## 11. 技术约束

- 保持 iOS 18 最低部署目标
- 保持 Swift 6 编译通过
- 保持 CocoaPods workspace 构建方式
- 不引入新的第三方分析 SDK
- 不依赖网络运行单元测试
- 手工代码编辑使用 `project.yml` 作为 XcodeGen 配置来源
- 不在本版本中大规模迁移现有 SwiftData schema
- 新增持久化字段时必须考虑旧 TestFlight 数据兼容

## 15. 风险与应对

### 风险 1：范围膨胀

本版本容易顺带进入上下文、多动作和智能排期改造。

应对：严格遵守“非目标”，新需求记录到后续版本，不在 `0.3.0` 实现。

### 风险 2：测试 EventKit 困难

直接测试真实 EventKit 会影响用户数据，也不适合自动运行。

应对：分离纯业务逻辑和系统写入，自动测试计划、状态和文案，真实写入由专用真机清单验证。

### 风险 3：诊断数据泄露隐私

原始模型响应可能包含聊天内容和任务信息。

应对：默认只保存脱敏元数据，完整内容必须显式开启，并且永远过滤凭证。

### 风险 4：SwiftData schema 变化导致旧数据丢失

新增诊断或批量结果字段可能触发持久化兼容问题。

应对：优先使用独立 Codable 存储或兼容默认值；任何 schema 变化必须使用旧版 TestFlight 数据升级验证。

### 风险 5：批量执行无法真正事务化

EventKit 逐条写入无法保证跨多条任务原子提交。

应对：本版本采用尽力执行和完整汇报，不伪装成事务；通用回滚与撤销留到 `0.6.0`。
