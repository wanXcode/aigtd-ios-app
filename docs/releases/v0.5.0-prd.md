## 2. 版本定位

`0.5.0` 的目标是把 AIGTD 从“模型一次输出一个 Action，客户端执行一次”升级为“模型可以按需查询任务、读取真实结果、继续决策，并在一个用户目标下完成多个相关操作”的本地任务 Agent。

本版本不是简单地把 `action` 改成数组。真正的升级包括：

1. 用强类型工具替代 View 内的大型动作分支
2. 允许模型先调用只读工具，再根据结果继续推理
3. 支持一个目标包含多个有序操作
4. 每个工具结果都来自真实 EventKit，并回到模型与 UI
5. 为循环次数、超时、取消、幂等和部分失败建立统一约束

`0.4.0` 已解决上下文、稳定引用、长期记忆与隐私问题。`0.5.0` 在这套上下文契约之上增加工具闭环，不重新发明任务引用，也不以继续扩张关键词规则作为主要实现方式。

## 3. 当前基线

`v0.4.0` 已具备：

- 统一的 `AgentContextSnapshot`
- 最近任务与展示序号的稳定 Reminder ID 引用
- 单任务创建、改期、移动、完成、删除和查看
- 重排方案预览与批量应用
- 会话摘要、长期偏好与隐私过滤
- 诊断阶段、Action 卡片和 96 项真机自动测试
- 100 条原始对话评测与 50 条上下文评测 fixture

当前执行链路仍有以下限制：

- `AgentRuntimeService` 每轮只返回一个 `MockAgentResult`
- 远端协议只允许单个 `action`
- EventKit 动作执行集中在 `ChatHomeView.executeResultAction`
- 模型无法在拿到真实查询结果后继续决策
- 一句话中的第二个动作通常被忽略或只能写进自然语言回复
- 不同动作的 pending、成功、部分失败和最终回复缺少统一运行状态
- 本地解析结果与远端结构化结果仍需要复杂仲裁

## 4. 核心问题

### 4.1 模型只能看到预先发送的快照

上下文快照有数量与隐私预算，不能包含全部 Reminders。模型遇到“找出下周到期的项目任务并移动到工作清单”时，可能需要先查询，再决定操作目标。

### 4.2 单 Action 无法表达完整用户目标

“把会议改到三点，再提醒我带资料”包含修改和创建两个动作。当前协议最多执行其中一个，另一个只能停留在文字承诺。

### 4.3 执行逻辑与 UI 耦合

`ChatHomeView` 同时负责模型调用、协议仲裁、ActionLog、EventKit 执行、引用回写、文案和卡片状态。继续增加动作会放大分支复杂度，难以单元测试和复用。

### 4.4 工具结果无法驱动下一步推理

当前执行结果只用于最终文案和日志。模型不能读取“找到 3 条同名任务”或“第二个动作失败”的结构化结果后选择澄清、调整参数或结束。

### 4.5 多动作放大安全与一致性风险

如果缺少幂等键、次数上限和确认边界，网络重试可能重复创建任务，部分失败可能让 UI 宣称全部成功，工具循环也可能失控。

## 5. 版本目标

### 5.1 产品目标

- AI 可以先查询任务，再依据真实查询结果继续处理
- 一条自然语言请求可以完成多个相关操作
- 多动作在执行前展示清晰计划，执行后展示逐项真实结果
- 同名、失效 ID 和缺失参数仍然停止执行并请求澄清
- 普通聊天与只读查询保持自然，不显示多余执行卡片
- 最终回复只陈述真实完成的操作，不把计划当作结果

### 5.2 工程目标

- 新增独立于 SwiftUI 的 `AgentOrchestrator`
- 建立版本化、强类型、可 Codable 的工具协议
- 将 EventKit 工具执行从 `ChatHomeView` 抽离
- 每个工具调用拥有稳定 `run_id`、`call_id` 和幂等键
- 统一表示成功、部分成功、失败、取消、超时和需要澄清
- 工具调用、结果与模型再推理形成可诊断状态机
- 保持 `v0.4.0` 数据模型和历史 ActionLog 可读取

### 5.3 成功指标

- 0.4.0 的 96 项自动测试全部保持通过
- 新增不少于 60 项工具、编排与多动作测试
- 新增不少于 40 条 0.5.0 多工具评测样例
- 发布场景中的多动作真实执行成功率达到 95% 以上
- 所有工具循环 100% 在配置上限内结束
- 重复投递相同 `call_id` 不产生重复写入
- 同名破坏性操作 100% 不静默选错
- 最终回复与 EventKit 真实结果一致率达到 100%

## 6. 非目标

以下能力不属于 `0.5.0`：

- 不实现任意第三方工具或插件市场
- 不接入日历、邮件、通讯录和网络搜索工具
- 不实现完整自然语言确认与通用撤销；这些进入 `0.6.0`
- 不实现复杂失败自动修复循环，最多允许一次参数纠正
- 不实现跨多个用户请求持续运行的后台 Agent
- 不实现 AI 主导的优先级、工时、依赖和日程优化；这些进入 `0.7.0`
- 不允许模型绕过本地安全策略直接调用 EventKit
- 不以新增大量中文关键词作为多动作理解的主要方案

## 7. 核心用户场景

### 7.1 查询后修改

```text
用户：把下周一的开会改到下午 3 点
```

期望流程：

1. 模型调用 `search_reminders`
2. 工具返回稳定 ID、标题、时间和清单
3. 唯一匹配时模型调用 `update_reminder`
4. EventKit 返回真实修改结果
5. 模型生成最终回复

若查询返回多个合理候选，停止写入并澄清。

### 7.2 两个连续动作

```text
用户：把会议改到下午 3 点，再提醒我 2 点半带资料
```

期望：

- 修改原会议时间
- 创建“带资料”提醒
- 两个动作使用同一运行计划
- 卡片分别显示每一步结果
- 只有两步都成功时才能回复“都处理好了”

### 7.3 批量移动与改期

```text
用户：把刚才列出的三条移到工作清单，并依次安排到本周三到周五
```

期望：

- 使用最近展示的稳定 ID 集合
- 先生成完整操作计划，不立即修改
- 用户点击确认后按顺序执行
- 单项失败不阻止安全的后续项
- 最终显示成功数、失败数和失败原因

### 7.4 查询后只回复

```text
用户：项目清单里哪几条没有时间？
```

期望：

- 调用 `search_reminders`
- 不调用任何写工具
- 不显示执行卡片
- 直接用真实查询结果回答

### 7.5 依赖前一步结果

```text
用户：新建一个“发布准备”清单，再把这两条移进去
```

期望：

- 第一步创建清单
- 第二步使用第一步返回的稳定清单 ID 或名称
- 第一步失败时不执行依赖它的移动操作
- 最终说明实际完成范围

### 7.6 部分失败

```text
用户：把这三条都标记完成
```

若其中一条已被系统 Reminders 删除：

- 其余可定位任务继续执行
- 被删除任务记录为 `not_found`
- 结果卡片显示部分完成
- 最终回复不得声称全部完成

## 8. 总体架构

```text
ChatHomeView
  -> AgentConversationCoordinator (@MainActor)
      -> AgentContextBuilder
      -> AgentOrchestrator
          -> AgentModelClient
          -> AgentToolRegistry
              -> ReminderQueryToolExecutor
              -> ReminderWriteToolExecutor
              -> ScheduleToolExecutor
          -> ToolExecutionLedger
      -> AgentRunPresenter
  -> SwiftData / ActionLog compatibility
  -> EventKit via ReminderStoreGateway
```

### 8.1 `AgentConversationCoordinator`

负责单次发送的 UI 生命周期：

- 创建用户消息和助手占位消息
- 构建上下文
- 启动、取消和恢复当前 run
- 将编排状态映射到消息与卡片
- 完成后更新会话摘要和稳定引用

它不直接解析工具参数，也不直接调用 EventKit。

### 8.2 `AgentOrchestrator`

负责：

- 调用模型获取下一步决策
- 验证工具名、参数、风险和调用预算
- 执行可执行工具
- 把结构化工具结果加入下一轮模型输入
- 在最终回复、澄清、确认、失败、取消或预算耗尽时结束

### 8.3 `AgentModelClient`

从当前 `AgentRuntimeService` 抽取模型传输职责：

- Chat Completions 与 Responses API 适配
- 请求、流式事件和错误解析
- 模型决策 JSON 解码
- 不负责本地 intent 仲裁或 EventKit 执行

### 8.4 `AgentToolRegistry`

注册允许调用的工具及其 schema、风险等级和执行器。未知工具必须拒绝，不允许动态反射调用任意方法。

### 8.5 `ReminderStoreGateway`

将 EventKit 读写包装为可替换协议。生产环境使用串行化访问的 `EventKitReminderStoreGateway` actor，测试使用内存实现。

Gateway 必须：

- 明确区分 `canRead` 与 `canWrite`，iOS 17 的 `writeOnly` 不得用于查询或引用解析
- 支持直接按稳定 ID 获取任务，不受上下文快照 40 条或列表刷新 50 条上限影响
- 写入前从 EventKit 重读当前状态并校验前置条件，避免快照与执行之间的状态变化
- 将 EventKit 错误映射为稳定错误分类，不向上层暴露框架细节

## 9. 编排状态机

```text
idle
  -> preparing_context
  -> deciding
  -> executing_reads
  -> deciding
  -> awaiting_confirmation
  -> executing_writes
  -> synthesizing
  -> succeeded | partial | failed | cancelled
```

状态规则：

- 同一 run 同时只能处于一个主状态
- 只读工具可以直接执行
- 多写操作必须先形成计划并等待按钮确认
- 单个低风险写操作保持 0.4.0 的直接执行体验
- 删除继续服从已保存事务规则和现有本地确认逻辑
- 进入终态后不得再次执行新工具
- App 退出或任务取消时停止未开始的调用

## 10. 模型决策协议

### 10.1 决策信封

```json
{
  "schema_version": 1,
  "run_id": "uuid",
  "goal": "调整会议并增加准备提醒",
  "phase": "tool_calls",
  "assistant_draft": "我先查一下会议，再一起处理。",
  "tool_calls": [
    {
      "call_id": "call-1",
      "tool": "search_reminders",
      "arguments": {
        "query": "会议",
        "date_scope": "next_7_days",
        "include_completed": false
      }
    }
  ],
  "final_reply": null
}
```

`phase` 只允许：

- `tool_calls`
- `awaiting_clarification`
- `awaiting_confirmation`
- `final`

### 10.2 工具结果信封

```json
{
  "schema_version": 1,
  "run_id": "uuid",
  "call_id": "call-1",
  "tool": "search_reminders",
  "status": "success",
  "result": {
    "items": [
      {
        "reminder_id": "stable-id",
        "title": "项目会议",
        "due_date": "2026-07-20T10:00:00+08:00",
        "list_id": "list-id",
        "list_title": "项目"
      }
    ]
  },
  "error": null
}
```

### 10.3 最终回复

模型只有在 `phase = final` 时提供 `final_reply`。客户端仍会校验：

- 回复不能声称未成功的操作已完成
- 部分失败时必须包含失败范围
- 无写工具成功时不能使用“已修改、已删除、已创建”等完成语气
- 模型最终回复不可用时，由本地结果生成确定性兜底文案

## 11. 工具目录

### 11.1 `search_reminders`

用途：按标题、清单、日期范围、完成状态筛选任务。

关键参数：

- `query`
- `list_id` / `list_title`
- `date_from` / `date_to`
- `include_completed`
- `limit`，最大 50

返回稳定 ID 和隐私允许的字段。

### 11.2 `get_reminder_details`

用途：读取一个或多个稳定 ID 的详情。

约束：

- 最多 10 个 ID
- 备注与已完成状态服从隐私开关
- 不允许标题模糊查询替代稳定 ID

### 11.3 `create_reminder`

用途：创建单条提醒事项。

参数：标题、日期、是否含时刻、目标清单、备注。

返回新 Reminder ID、最终标题、清单和到期时间。

### 11.4 `update_reminder`

用途：按稳定 ID 修改标题、备注或时间。

约束：

- 必须提供稳定 ID
- 只更新显式提供的字段
- 日期型和日期时间型保持 `includes_time`

### 11.5 `move_reminder`

用途：按稳定 ID 移动清单。

目标优先使用 `list_id`，名称仅用于用户可读展示和严格匹配。

### 11.6 `complete_reminder`

用途：按稳定 ID 设置完成或恢复未完成。

必须返回修改后的真实完成状态。

### 11.7 `delete_reminder`

用途：按稳定 ID 删除提醒事项。

约束：

- 仅允许稳定 ID
- 保持删除确认事务规则
- 删除成功后立即使相关引用失效

### 11.8 `propose_schedule`

用途：根据指定稳定 ID 集合生成计划，不写 EventKit。

返回确定性 `plan_id` 与计划项。

### 11.9 `apply_schedule`

用途：应用已确认的 `plan_id`。

约束：

- 只能应用当前会话中未过期计划
- 每项按稳定 ID 执行
- 返回完整批量报告

## 12. 工具调用规则

### 12.1 调用预算

- 单个 run 最多 4 轮模型决策
- 单个 run 最多 8 次工具调用
- 单轮最多 5 个并列工具调用
- 写工具总数超过 1 时必须先确认
- 达到预算后停止并返回可理解错误，不继续重试

### 12.2 顺序与依赖

- 默认按模型返回顺序执行
- 只读且互不依赖的调用可在后续优化中并发；首版保持串行以便诊断
- 依赖前一结果的调用必须放在下一轮模型决策中
- 前置调用失败时，依赖项不得执行

### 12.3 幂等

幂等键：`run_id + call_id`。

- 同一键重复提交返回已有结果
- `create_reminder` 不得因网络重试重复创建
- 已完成的工具调用不可在恢复 run 时再次执行
- 幂等账本至少保留 24 小时
- 目标已处于期望状态时返回 `unchanged`，不重复改写完成时间或到期时间
- 已成功删除的调用重放时返回 `already_applied`，不降级成新的 `not_found` 失败

### 12.4 写入前置条件

每个写调用可以携带：

- `expected_list_id`
- `expected_due_date`
- `expected_completion`
- `must_exist`

执行器必须在真正写入前按稳定 ID 重读 EventKit。若当前状态与模型决策时的前置条件冲突，返回 `precondition_conflict`，不得基于旧快照继续覆盖用户在系统 Reminders 中的修改。

### 12.5 超时与取消

- 只读工具默认超时 8 秒
- 单写工具默认超时 12 秒
- 批量计划默认超时 30 秒
- 用户取消后停止未开始调用
- 已提交给 EventKit 的单次原子写入不伪装成已撤销

## 13. 安全策略

### 13.1 风险等级

| 等级 | 工具 | 默认策略 |
| --- | --- | --- |
| 只读 | search、details、propose | 直接执行 |
| 低风险写 | create、单条 update、move、complete | 唯一目标时直接执行 |
| 中风险写 | 多条 update、move、complete、apply_schedule | 展示计划并确认 |
| 高风险写 | delete、批量 delete | 服从事务规则；批量始终确认 |

### 13.2 本地强制校验

模型不能覆盖以下规则：

- 稳定 ID 必须存在
- ID 与显式标题、日期、清单约束冲突时停止执行
- 同名目标不唯一时停止执行
- 未授权备注与已完成任务不得进入工具结果
- 计划过期或上下文已变更时重新确认
- 未知工具、未知字段和超预算调用拒绝执行
- 写工具参数必须通过本地日期、清单和字段校验
- 查询不得受 UI 快照数量上限影响；稳定 ID 查询必须直接访问 Store Gateway

### 13.3 本地规则边界

本地规则只负责：

- 安全策略
- 参数类型与范围校验
- 稳定引用解析
- 无网络时的 0.4.0 单动作兼容兜底
- 确定性结果文案

开放语义理解、多动作拆解与查询策略由模型负责。

## 14. 执行结果模型

每次工具调用生成 `AgentToolInvocation`：

- `id`
- `runID`
- `callID`
- `toolName`
- `riskLevel`
- `argumentsJSON`
- `status`
- `resultJSON`
- `errorCategory`
- `userVisibleError`
- `startedAt`
- `finishedAt`
- `idempotencyKey`

状态只允许：

- `queued`
- `running`
- `awaiting_confirmation`
- `success`
- `failed`
- `skipped`
- `cancelled`
- `timed_out`
- `unchanged`
- `already_applied`

整个 run 的状态由调用结果归并，不由模型自由填写。

## 15. 数据与迁移

### 15.1 新增持久化对象

- `AgentRunLog`
- `AgentToolInvocation`
- `PendingOperationPlan`

### 15.2 兼容策略

- 现有 `ActionLog` 保留读取和展示能力
- 0.4.0 历史消息无需迁移成 ToolInvocation
- 新 run 可以为兼容 UI 生成一个汇总 ActionLog，但真实状态以 run 和 invocation 为准
- schema 损坏时只清理编排运行数据，不删除聊天、设置或 Reminders

## 16. UI 与交互

### 16.1 消息气泡

- 流式阶段只显示自然语言草稿，不显示 JSON 或工具参数
- 只读查询完成后直接替换为最终答案
- 等待工具时显示简短、即时的状态，不重复输出多条助手消息

### 16.2 多动作卡片

一张卡片代表一个用户目标，内部逐项显示：

- 操作名称
- 目标任务
- 计划变更
- 当前状态
- 失败原因

卡片整体状态：待确认、执行中、已完成、部分完成、失败、已取消。

### 16.3 确认

- 多写操作显示“执行这个计划”按钮
- 确认前不写 EventKit
- 用户修改原请求时废弃旧计划并生成新计划
- 自然语言“执行吧”留到 0.6.0；0.5.0 以按钮确认为准

### 16.4 过渡

- 第一段可见回复后立即显示稳定的执行状态
- 卡片不突然消失；完成后在原位置更新
- 多项结果使用渐进状态，不通过插入和删除整张卡片制造跳动

## 17. 诊断与可观测性

- `orchestrator_started`
- `model_decision_received`
- `tool_validation_completed`
- `tool_execution_started`
- `tool_execution_completed`
- `tool_result_appended`
- `model_reentry_started`

- 新协议解析失败时不得执行模型自然语言中暗示的写操作

## 21. 发布验收用例

### 用例 1：查询后修改

输入：`把下周一的开会改到下午 3 点`

通过标准：先查询并唯一定位，再按 ID 修改；同名时澄清。

### 用例 2：修改加创建

输入：`把会议改到下午 3 点，再提醒我 2 点半带资料`

通过标准：两个动作均真实执行，标题和时间正确，结果卡片逐项成功。

### 用例 3：创建清单后移动

输入：`新建发布准备清单，把刚才两条移进去`

通过标准：创建成功后移动；创建失败时移动被跳过。

### 用例 4：只读详情

输入：`项目清单里哪些任务没有时间？`

通过标准：只查询、不写入、不显示执行卡片。

### 用例 5：批量计划

输入：`把刚才三条移到工作清单，并安排到本周三到周五`

通过标准：先显示计划；确认前不修改；确认后逐项真实执行。

### 用例 6：部分失败

准备：计划生成后在系统 Reminders 删除其中一条。

通过标准：其余项继续，删除项报告 `not_found`，最终状态为部分完成。

### 用例 7：幂等

模拟同一 `call_id` 重复投递创建工具。

通过标准：只创建一条提醒事项，第二次返回已有结果。

### 用例 8：循环保护

模拟模型连续请求无效工具。

通过标准：达到预算后停止，不写 EventKit，并给出可理解错误。

### 用例 9：隐私

关闭备注和已完成任务开关后请求详情。

通过标准：工具结果和诊断均不包含受保护内容。

### 用例 10：普通聊天

输入：`我今天有点累`

通过标准：自然回复，无工具调用、无卡片、无 Reminders 修改。

### 用例 11：外部修改冲突

准备：生成多动作计划后，在系统 Reminders 手动修改其中一条的时间或清单。

通过标准：确认执行时该项返回 `precondition_conflict`，不覆盖用户的新值；其他独立项按 best-effort 规则继续。

### 用例 12：权限能力

准备：模拟 full access、write-only 和 denied 三种权限。

通过标准：full access 可查询和写入；write-only 只允许支持的创建写入，不得假装已读取任务；denied 不执行任何工具。

## 23. 风险与应对

### 风险 1：模型循环失控

应对：固定轮次和调用预算；达到上限立即结束；未知工具不重试。

### 风险 2：重复创建或重复写入

应对：持久化幂等账本；所有写工具使用 `run_id + call_id`。

### 风险 3：多动作部分成功造成误导

应对：结果由本地归并；最终回复经过真实性校验；卡片逐项展示。

### 风险 4：ChatHomeView 重构引入回归

应对：先抽工具层并保持旧入口，再切换 Orchestrator；每阶段运行完整回归。

### 风险 5：本地规则与模型继续冲突

应对：本地只保留安全、类型、稳定引用和兜底；合法远端工具决策不再被关键词 intent 覆盖。

### 风险 6：工具结果泄露隐私

应对：所有读取工具只能使用隐私过滤后的查询服务；诊断独立脱敏。

### 风险 7：快照与写入之间被外部修改

应对：写工具携带前置条件，并在 EventKit mutation 前按稳定 ID 重读校验；冲突时停止该项。

### 风险 8：权限能力判断错误

应对：统一权限能力模型，严格区分读写；`writeOnly` 永远不能满足查询工具的授权要求。

## 24. Definition of Done

`0.5.0` 完成必须满足：

- `AgentOrchestrator` 成为远端工具调用的唯一编排入口
- EventKit 写入不再由 `ChatHomeView` 直接分支执行
- 九个本地工具拥有强类型 schema 和独立测试
- 模型可以基于真实查询结果继续至少一轮决策
- 一个用户目标可以执行多个有序工具调用
- 多写操作确认前不修改 EventKit
- 幂等、预算、超时、取消和部分失败均有自动测试
- 最终回复与真实工具结果一致
- 0.4.0 的 96 项测试和既有评测无回退
- 新增工具与多动作测试、评测达到门槛
- 从 `0.4.0 (17)` 升级数据安全
- TestFlight 真机验收通过

满足以上条件后，项目进入 `0.6.0` 的自然确认、撤销和失败恢复开发阶段。
