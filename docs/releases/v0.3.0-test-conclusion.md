# AIGTD 0.3.0 测试结论

测试日期：2026-07-13
初测版本：`0.3.0 (5)`
修复候选版本：`0.3.0 (13)`
测试方式：iPhone 真机手动验收

## 总体结论

`0.3.0 (13)` 修复候选版本的核心真机功能验收通过。初测发现的单任务改期、重排方案执行、歧义删除保护、普通聊天卡片、卡片过渡、同步文案和诊断状态问题均已修复并完成复测。

## 初测结果

| 测试项 | 结果 | 现象 | 风险 |
| --- | --- | --- | --- |
| 用例 2：单任务改期 | 失败 | “改到后天上午 10 点”被误判为批量重排，未修改原任务时间 | 高 |
| 用例 4：连续上下文与改期 | 失败 | 能识别目标任务并生成 `plan_reschedule` 方案，但方案无法执行，系统提醒事项时间未变化 | 高 |
| 用例 7：同名任务删除 | 失败 | 两条同名任务同时存在时没有要求澄清，直接删除了明天下午的任务 | 严重 |
| 普通聊天不执行 Action | 通过 | 普通聊天没有真实创建、修改或删除提醒事项 | 低 |
| 普通聊天执行卡片 | 失败 | 普通聊天期间错误闪现“正在记”执行卡片 | 中 |
| 执行卡片过渡 | 失败 | 临时卡片在请求结束后突然消失，缺少自然的状态过渡 | 中 |
| Reminders 同步文案 | 失败 | 显示“最新同步 0 秒后”，正确应为“最新同步 0 秒前”或“刚刚同步” | 低 |

## 修复候选版本复测

- 用例 6“任务不存在时修改”：通过。回复与卡片均显示未找到目标，未误报成功。
- Reminders 同步文案：通过。同步后先显示“刚刚同步”，随后按秒更新为“最新同步 x 秒前”。
- Agent 诊断页：通过。记录可进入列表与阶段详情；普通聊天最终显示“已完成”和“无任务动作”，详情以“完成回复”结束，未显示完整原文或凭证。

## 最终复测结论

- 用例 2：通过。上下文单任务改期真实写入指定时间，不再误判为批量重排。
- 用例 4：通过。连续上下文中的任务移动和改期能够真实执行。
- 用例 5：通过。批量重排方案可先预览再应用，并显示总数、成功数和失败数。
- 用例 6：通过。目标不存在时明确失败，不创建错误任务。
- 用例 7：通过。同名任务存在歧义时停止删除并要求澄清。
- 用例 8：通过。普通聊天不执行任务，也不闪现执行卡片。
- 用例 9：通过。清单顺序、空清单、浅色页面和同步时间文案符合预期。
- 用例 10：通过。诊断记录可查看阶段状态，且默认不泄露完整用户内容和凭证。

当前结论：核心真机手动测试通过，可以进入 Release 构建、自动测试和 TestFlight 发布检查。

## 已确认通过

- 创建带日期和清单的提醒事项可以真实写入系统 Reminders。
- 删除目标明确的提醒事项可以真实执行。
- 连续对话中的“把它放到项目清单”能指向刚创建的任务并完成移动。
- 普通聊天最终不会修改系统提醒事项数据。
- Reminders 页面保持浅色风格。
- App 清单顺序与系统 Reminders 清单顺序一致。
- 没有任务的空清单仍会显示。

## 已解除的发布阻断项

1. 单任务时间修改已使用独立、可执行的更新动作，不再误入批量重排流程。
2. `plan_reschedule` 方案已能在用户点击应用后真实更新系统提醒事项，并正确回写结果状态。
3. 删除操作存在多个候选任务时会停止执行并要求用户澄清，不再默认选择候选项。

## 修复后复测标准

- 输入“改到后天上午 10 点”时，只修改最近上下文中的同一条任务，不创建重复任务，不展示批量重排方案。
- 生成单条或批量重排方案后，点击“应用这个方案”能够真实更新时间，并将卡片稳定更新为最终状态。
- 两条同名任务同时存在时，输入“删除开会”必须列出或描述候选项并要求澄清，系统数据保持不变。
- 普通聊天从开始到结束都不展示 Action 执行卡片；若最终意图需要执行，卡片只在结构化意图确认后出现。
- 临时卡片到最终卡片应保持同一视图身份并平滑更新，不应突然消失或闪现错误动作。
- 同步刚完成时显示“刚刚同步”；其他时间只显示过去式文案，不得出现“秒后”。
