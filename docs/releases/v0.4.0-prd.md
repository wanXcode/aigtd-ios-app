## 2. 版本定位

`0.4.0` 的目标是让 AIGTD 在执行单个任务动作前，真正知道用户当前有哪些任务、最近讨论了什么、代词指向哪一条任务，以及哪些偏好是用户明确确认过的。

本版本不是继续堆叠关键词规则，也不引入多工具循环。它将在 `0.3.0` 已稳定的单 Action 执行链路前增加统一的上下文构建层，并将执行结果中的稳定任务 ID 带回后续对话。

本版本聚焦五件事：

1. 建立统一、结构化、可测试的 `AgentContextSnapshot`
2. 用稳定 Reminder ID 解析“刚才那条、第二条、上一条”等指代
3. 在最近 8 轮窗口之外维护受控的会话摘要
4. 正式接通 Prompt、Solu、Memory 和 Operating Guide
5. 明确任务数据发送范围、记忆写入条件和隐私控制

## 3. 当前基线

`v0.3.0` 已具备以下基础：

- Chat 会把最近 8 条非流式消息发送给模型
- Prompt 会附带最多 8 条未完成任务的标题和清单
- 创建任务成功后，`ActionLog.undoToken` 保存 EventKit Reminder ID
- “刚才创建的任务”可在部分操作中使用最近创建 ID
- Agent 页面已经存在 Prompt、Memory、Solu 和 Operating Guide 四类文档
- Runtime 当前实际使用 Memory 和 Operating Guide，但 Prompt 与 Solu 尚未完整进入系统提示词
- 诊断链路、100 条评测集和 28 项自动测试已经建立

当前能力仍然是分散的：上下文拼装位于 `AgentRuntimeService`，最近任务引用位于 `ChatHomeView`，会话数据位于 SwiftData，长期偏好位于自由文本 Memory。它们没有统一的数据契约、生命周期和隐私规则。

## 4. 核心问题

### 4.1 任务上下文不完整

模型只看到少量标题预览，不包含稳定 ID、到期时间、完成状态和明确的相关性来源。模型可能知道“有一条开会”，但不知道用户指的是哪一个 EventKit 对象。

### 4.2 指代解析依赖标题和局部特例

目前仅对“最近创建任务”有局部 ID 解析。最近修改、移动、展示、选择过的任务没有统一引用记录；“第二条”也无法稳定映射到刚展示的结果集合。

### 4.3 长对话会遗忘

最近消息窗口固定为 8 条。超过窗口后，当前讨论目标、已经确认的约束和最近展示的任务集合都会丢失。

### 4.4 Agent 文档没有完整生效

四类文档已经可编辑，但 Runtime 只显式使用部分内容。用户修改 Prompt 或 Solu 后，下一轮行为不一定发生可验证的变化。

### 4.5 Memory 缺少写入边界

当前 Memory 是自由文本，没有区分系统默认、用户明确偏好和普通聊天内容。若直接自动总结并写入，容易保存错误信息或私人内容。

### 4.6 隐私范围不可见

用户无法明确知道每轮会向远端模型发送哪些任务字段，也不能控制是否发送备注、已完成任务或更多历史对话。

## 5. 版本目标

### 5.1 产品目标

- 连续对话中稳定理解最近创建、修改、移动、展示和选择过的任务
- 同名任务优先通过稳定 ID、展示顺序和时间条件解析，不静默猜测
- 长对话超过最近窗口后仍保留当前讨论目标和已确认约束
- Agent 文档修改后在下一轮请求中真实生效
- 只保存用户明确表达的稳定偏好，不把普通聊天自动写入长期记忆
- 用户可以查看和控制发送给模型的任务数据范围

### 5.2 工程目标

- 上下文构建从 View 和 Prompt 字符串拼接中独立出来
- 上下文、引用解析和记忆策略均可离线单元测试
- 所有上下文结构带 schema 版本，支持后续迁移或安全清空
- 单次请求的上下文有明确大小预算和截断顺序
- 诊断只记录上下文结构、数量和哈希，默认不保存完整内容

### 5.3 成功指标

- `v0.3.0` 的 28 项自动测试保持全绿
- 100 条原始评测集不降低既有通过率
- 新增不少于 50 条上下文与记忆评测样例
- 稳定 ID 指代评测通过率达到 95% 以上
- 所有破坏性同名任务场景 100% 不静默选错
- 20 轮连续对话验收场景中，当前目标和约束保持正确
- Prompt、Solu、Memory、Operating Guide 的生效均有自动测试
- 隐私与凭证测试 100% 通过

## 6. 非目标

以下能力不属于 `0.4.0`：

- 不实现模型查询任务后再次推理的多轮工具调用
- 不支持一次回复执行多个不同 Action
- 不实现 `AgentOrchestrator` 工具循环
- 不实现自然语言确认、通用撤销和失败恢复工作流
- 不实现 AI 主导的优先级、工时、依赖和日程规划
- 不建立跨设备云端长期记忆服务
- 不自动保存完整聊天记录为长期记忆
- 不让模型直接决定要永久保存哪些隐私信息
- 不以新增大量本地关键词作为主要交付方式

多工具与多动作进入 `0.5.0`，自然确认与撤销进入 `0.6.0`，智能规划进入 `0.7.0`。

## 7. 用户场景

### 7.1 最近创建任务

```text
用户：明天下午 3 点提醒我开会
用户：把刚才那条移到项目清单
用户：再改到后天上午 10 点
```

期望：

- 三轮始终引用同一个 Reminder ID
- 后两轮不依赖“开会”标题模糊搜索
- 卡片和系统 Reminders 最终状态一致

### 7.2 展示结果中的序号引用

```text
用户：看看明天的任务
AIGTD：明天有 3 条……
用户：把第二条改到下午 4 点
```

期望：

- 查看结果保存有序 ID 集合
- “第二条”解析为刚展示集合中的第 2 个 ID
- 若集合已经失效或目标被删除，停止执行并说明

### 7.3 同名任务

```text
用户：删除开会
```

期望：

- 若上下文中有唯一被选中的“开会”，可使用其稳定 ID
- 若没有唯一上下文引用且存在多个候选，必须澄清
- 不因某条标题精确匹配就绕过其他合理候选

### 7.4 长对话

用户先讨论任务范围和时间约束，随后经过 10 轮以上补充，再说“按刚才的范围重新排”。

期望：

- 会话摘要仍包含当前目标、范围和已确认约束
- 摘要不保存模型内部推理过程
- 未确认的建议不能被写成用户偏好

### 7.5 明确长期偏好

```text
用户：以后没有具体时间的任务默认放上午 9 点
```

期望：

- 识别为稳定偏好候选
- 明确展示将保存的偏好内容
- 只有用户明确表达“以后、默认、总是、记住”等长期语义时才可写入
- 用户可以在 Agent 页面查看、修改或删除

### 7.6 普通聊天

```text
用户：我今天有点累，不想整理任务
```

期望：

- 可以自然回复
- 不创建任务，不改变上下文目标
- 不写入长期 Memory

## 8. 上下文数据契约

### 8.1 `AgentContextSnapshot`

新增纯 Swift、`Codable`、`Sendable` 的版本化结构：

```swift
struct AgentContextSnapshot: Codable, Sendable {
    let schemaVersion: Int
    let generatedAt: Date
    let timeZoneIdentifier: String
    let session: SessionContext
    let recentTurns: [AgentConversationTurn]
    let sessionSummary: SessionSummary?
    let reminders: [ReminderContextItem]
    let references: ReferenceContext
    let preferences: [UserMemoryItem]
    let documents: AgentDocumentContext
    let privacy: ContextPrivacyDescriptor
}
```

要求：

- `schemaVersion` 首版为 `1`
- Snapshot 是单轮请求的不可变值，不在执行中原地修改
- 任何截断都必须在 `privacy` 或诊断元数据中记录数量
- Prompt 只消费 Snapshot，不再自行查询 View 状态

### 8.2 `ReminderContextItem`

默认字段：

- `id`：EventKit calendar item identifier
- `title`
- `listID` 与 `listTitle`
- `dueDate`
- `isCompleted`
- `lastModifiedAt`，若 EventKit 可提供
- `relevanceReason`：recently_created、recently_modified、recently_shown、date_scope、list_scope 等

备注字段默认不发送。只有用户在隐私设置中开启“允许 AI 读取任务备注”后，才可包含经过长度限制的 `notesPreview`。

### 8.3 `ReferenceContext`

保存：

- 最近创建任务 ID
- 最近修改任务 ID
- 最近移动任务 ID
- 最近完成任务 ID
- 最近展示的有序任务 ID 集合
- 最近由用户明确选择的任务 ID
- 每个引用的来源消息 ID、记录时间和失效状态

引用不保存任务内容副本；显示时以最新 Reminders Snapshot 为准。

### 8.4 `SessionSummary`

会话摘要是结构化状态，不是自由推理文本：

- 当前讨论目标
- 当前任务范围
- 已确认约束
- 待澄清问题
- 相关任务 ID
- 摘要覆盖到的最后消息 ID
- 更新时间

禁止保存：

- 模型思考过程
- 未经用户确认的推测
- API Key、语音凭证和完整 Authorization
- 与事务管理无关的私人闲聊原文

## 9. 上下文构建策略

### 9.1 新增 `AgentContextBuilder`

职责：

1. 读取当前会话状态
2. 根据缓存时效刷新 Reminders
3. 合并最近消息、会话摘要和引用记录
4. 按隐私设置裁剪任务字段
5. 按上下文预算排序和截断
6. 生成不可变 Snapshot

`ChatHomeView` 只负责触发构建并传递结果，不再负责拼装最近任务引用。

### 9.2 Reminders 刷新规则

- 请求开始时，若最近同步超过 5 秒，先刷新 Reminders
- 涉及“今天、明天、某清单、未完成、刚才那条、第二条”等任务语境时强制刷新
- 刷新失败时可使用最近一次快照，但必须标记 `isStale = true`
- 破坏性动作在执行前仍由 EventKit 再次确认目标存在
- 上下文刷新不得直接修改任务

本地预判只决定是否刷新上下文，不决定最终 Action intent。

### 9.3 相关任务排序

优先级从高到低：

1. 最近明确选择的任务
2. 最近创建、修改、移动或展示的任务
3. 用户当前指定日期或清单范围内的任务
4. 今天和已逾期任务
5. 其他未完成任务

默认最多发送 40 条任务。超过预算时先删除低相关任务，不截断稳定引用对应的任务。

### 9.4 上下文预算

- 最近原始对话：最多 8 条，每条最多 300 字符
- 会话摘要：最多 1,200 字符的结构化序列化结果
- Reminder 项：最多 40 条
- 单条标题：最多 120 字符
- 单条备注预览：最多 200 字符，默认关闭
- 每类 Agent 文档：最多 4,000 字符
- 超出预算时按固定顺序裁剪，并记录裁剪计数

## 10. 稳定引用解析

### 10.1 新增 `AgentReferenceResolver`

输入：

- 模型返回的 `target_id`、`target`、`ordinal`
- 当前 `ReferenceContext`
- 当前 Reminders Snapshot

输出：

- `resolved(identifier)`
- `ambiguous(candidates)`
- `notFound`
- `staleReference`

### 10.2 解析优先级

1. 有效 `target_id`
2. 最近明确选择 ID
3. “刚才、上一条、它”等上下文引用
4. “第一条、第二条”等最近展示集合序号
5. 标题、日期、清单联合匹配
6. 无法唯一确认时澄清

破坏性操作不得从失败的 ID 解析静默降级到宽松标题匹配。

### 10.3 Action 协议扩展

保持单 Action envelope，但为相关动作增加可选实体：

- `target_id`
- `target`
- `ordinal`
- `reference_source`
- `due_date`
- `destination_list`

模型应优先返回上下文中提供的 `target_id`。本地执行层必须验证 ID 对应任务仍存在。

## 11. 会话摘要

### 11.1 生成时机

- 会话有效消息超过 8 条后生成首个摘要
- 此后每新增 6 条有效消息增量更新
- 创建、修改、移动、完成、删除或展示任务后立即更新相关 ID 状态
- 清空会话时同步删除会话摘要和引用记录

### 11.2 生成方式

首版使用确定性的本地结构化归纳：

- 从成功 ActionLog 提取已执行事实
- 从查看结果提取最近展示集合
- 从用户明确表达提取当前目标和约束候选
- 不额外发起隐藏模型请求

如后续需要模型润色摘要，必须作为可关闭能力，并且不能保存模型推测为事实。

## 12. 长期记忆

### 12.1 可保存内容

- 用户称呼
- 默认时区
- 默认任务时间
- 默认清单偏好
- 明确的工作日或时间段偏好
- 用户明确要求“记住”的稳定事务规则

### 12.2 不可自动保存内容

- 普通聊天情绪
- 单次任务内容
- 未确认的模型推测
- 联系方式、地址、健康、财务等敏感信息
- 完整聊天原文
- 凭证和系统诊断内容

### 12.3 写入规则

- 只有用户明确表达长期语义时才生成记忆候选
- 写入内容必须是简短、可读、可编辑的规则
- 写入后在对话中明确告知保存了什么
- Memory 页面提供查看、编辑、删除和恢复默认值
- 删除记忆后，下一轮 Snapshot 不得再包含该条

本版本不让模型在没有用户长期语义的情况下自主写 Memory。

## 13. Agent 文档接通

四类文档全部进入 `AgentDocumentContext`：

- Prompt：角色、表达方式和最高层行为目标
- Solu：当前产品能力、工具边界和版本策略
- Memory：用户明确确认的长期偏好
- Operating Guide：执行原则和风险约束

要求：

- 文档修改后下一轮请求立即生效
- 诊断记录每类文档的版本哈希和长度，不记录默认完整内容
- 空文档回退到系统默认值
- 系统 JSON schema 和安全约束不可被用户文档覆盖
- 增加“恢复默认”操作

## 14. 隐私与设置

Agent 页面新增“上下文与隐私”区域：

- 当前会话上下文状态
- 最近引用任务数量
- 会话摘要更新时间
- 长期记忆条目数量
- 允许读取任务备注：默认关闭
- 允许包含已完成任务：默认关闭
- 发送任务上限：默认 40
- 清除会话上下文
- 清除长期记忆

页面需明确说明：标题、清单、时间等上下文可能发送给用户配置的远端模型服务商。

## 15. 存储与迁移

### 15.1 存储方案

首版使用独立、版本化的 Codable 存储：

- `AgentSessionContextStore`：按 session UUID 保存摘要和引用
- `AgentContextPrivacyStore`：保存隐私选项
- Memory 继续使用现有 `AgentDocument`

不为会话上下文新增 SwiftData `@Model`，避免在 0.4 首版引入不必要的持久化 schema 风险。

### 15.2 生命周期

- 会话上下文保留 30 天
- 删除 ChatSession 时删除对应上下文
- 引用任务不存在时标记失效，并在 7 天后清理
- 存储结构无法迁移时安全清空上下文，不影响聊天和 Reminders 数据
- 从 `v0.3.0` 升级时自动创建空的 v1 上下文，不改动现有消息和 Agent 文档

## 16. 诊断与可观测性

每次请求新增阶段：

- `context_refresh`
- `context_build`
- `reference_resolution`
- `session_summary_update`
- `memory_update`

默认只记录：

- Snapshot schema 版本
- 消息、任务、引用和记忆数量
- 是否使用陈旧快照
- 各类文档长度和哈希
- 截断数量
- 引用解析结果类型

默认不得记录完整标题、备注、会话摘要和 Memory 内容。

## 17. 交互要求

- 上下文解析成功时不增加机械说明，继续自然回复
- 使用“第二条”成功后，结果卡展示实际任务标题
- 引用失效时说明“刚才那条已经不存在或发生变化”
- 存在多个候选时列出足够区分的标题、时间和清单
- 保存长期偏好后用一句话明确告知，不显示内部 Memory schema
- 清除上下文或记忆属于本地数据操作，不删除 Reminders

### 18.1 单元测试

- Snapshot 构建、排序、预算和截断
- Reminders 陈旧快照策略
- 最近创建、修改、移动、展示、选择引用
- 序号引用与失效引用
- 同名任务破坏性操作保护
- 会话摘要增量更新
- Memory 写入白名单和敏感内容拒绝
- 四类 Agent 文档接通与默认回退
- Codable schema 迁移和损坏恢复
- 上下文诊断脱敏

### 18.2 评测集

在原 100 条基线上新增至少 50 条：

- 最近对象指代 15 条
- 展示集合序号 10 条
- 长对话摘要 10 条
- 同名任务安全 5 条
- 长期偏好写入与删除 5 条
- 隐私与错误恢复 5 条

### 18.3 真机测试

- 从 TestFlight `0.3.0 (16)` 升级，历史聊天和 Agent 文档保留
- 20 轮连续对话后指代仍正确
- 同名任务不误删、不误改
- 关闭备注权限后远端上下文不含备注
- 清除会话上下文后“刚才那条”不再使用旧引用
- 修改 Prompt 后下一轮回复行为发生可观察变化
- App 重启后会话摘要和引用可恢复

## 19. 验收用例

### P0

1. 创建任务后连续移动和改期，始终操作同一 Reminder ID
2. 查看明天任务后说“把第二条改到下午 4 点”，正确修改展示集合第 2 条
3. 两条同名任务存在时，没有唯一引用就停止破坏性操作

## 21. 风险与应对

### 风险 1：上下文过大导致成本和延迟上升

应对：固定预算、相关性排序、默认不发送备注，并在诊断中记录截断。

### 风险 2：陈旧 ID 指向错误对象

应对：执行前重新从 EventKit 验证 ID；ID 失效后不降级到破坏性模糊匹配。

### 风险 3：会话摘要把模型推测当成事实

应对：首版本地确定性归纳，只从用户明确表达和成功 ActionLog 提取事实。

### 风险 4：长期记忆保存过度

应对：只接受明确长期语义，敏感类别禁止自动写入，并提供可见管理与清除入口。

### 风险 5：本地规则继续膨胀

应对：本地逻辑只负责数据选择、安全验证和确定性引用，不承担开放语义理解；最终 intent 仍由模型结构化输出决定。

### 风险 6：0.4 范围滑向多工具 Agent

应对：严格保持单请求、单 Action。任何“先查再推理再执行”需求进入 0.5。

## 22. Definition of Done

`0.4.0` 完成必须满足：

- `AgentContextSnapshot` 成为远端与本地 fallback 的统一上下文输入
- View 不再单独维护业务引用解析逻辑
- 创建、修改、移动、完成、删除和查看结果都能回写稳定引用
- 最近对象和展示序号引用通过自动与真机测试
- 会话摘要跨越最近 8 条窗口并可在重启后恢复
- 四类 Agent 文档下一轮全部真实生效
- 长期 Memory 只保存用户明确的稳定偏好
- 隐私设置、清除入口和说明可用
- 原 28 项测试和 100 条评测基线无回退
- 新增上下文评测达到发布门槛
- 诊断默认不泄露任务内容、Memory 和凭证
- 从 `0.3.0 (16)` 升级数据安全
- TestFlight 真机验收通过

满足以上条件后，项目进入 `0.5.0` 的工具调用与多动作开发阶段。
