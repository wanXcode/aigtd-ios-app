# AIGTD Reminders QA 验收清单

## 目标

这份清单用于下一轮真机测试验收，重点检查：

- AIGTD 对话气质是否接近 `gtd-tasks` / 飞书版 AIGTD
- 建任务、看任务、测试句边界是否正确
- 真任务执行后是否真的落到 `Apple Reminders`
- 结果卡、查看类回复、试探句回复是否符合当前交互原则
- 单行输入区、键盘联动、发送按钮状态和语音 partial / final 回填是否接近飞书 IM / 豆包的体验

## 本轮必须覆盖的关键路径

### 1. 明确建任务

测试目标：

- 明确任务录入语句应直接创建 reminder
- 创建成功后应显示成功回执

建议输入：

- `帮我记一个任务：测试 ios 端`
- `帮我记个待办：整理报销`
- `明天提醒我给张闯回信`
- `下周一整理报销，先放未来`

验收点：

- 任务应真实进入系统 `Reminders`
- 聊天回复要像 AIGTD，不要像接口回执
- 成功时应出现结果卡
- 结果卡要说明任务、时间、清单，而不是技术字段

### 2. 明确查看任务

测试目标：

- 查询类输入应走查看分支，不应创建任务
- 回复应先给结论，再补少量细节

建议输入：

- `今天我还有什么事`
- `明天我有什么`
- `看看我现在有哪些事`
- `看看收集箱里有什么`

验收点：

- 不应创建 reminder
- 不应显示结果卡
- 回复应自然、简短、像秘书汇报
- 优先展示最重要的 2 到 3 条

### 3. 纯测试 / 试探句

测试目标：

- 试探系统是否在线的句子，不应误建任务
- 只保留自然回复

建议输入：

- `测试一下这个系统`
- `你在吗`
- `hello`
- `试一下`

验收点：

- 不应创建 reminder
- 不应显示结果卡
- 回复要自然，不要机械

### 4. 输入区与语音输入

测试目标：

- 单行输入区的键盘联动、首次聚焦、发送反馈和语音流式回填要顺
- 语音按钮只负责语音，不需要额外功能按钮干扰

建议输入与操作：

- 点击输入框，确认一次点击即可聚焦
- 点击语音按钮，确认进入录音态
- 说一句完整的话，确认 partial 文本会实时进输入框
- 停止录音，确认 final 文本会覆盖 partial 文本
- 点击发送，确认按钮状态不会明显卡顿

验收点：

- 键盘弹出后，消息列表应自然上移，不遮挡最后一条消息
- 输入框第一次点击就应聚焦，不应出现二次点击才生效
- 发送按钮在发起请求后应立即有反馈，不应长时间灰住
- 语音输入时，输入框应实时增长文字，而不是录完才一次性出现
- 结束语音后，最终文本应保留在输入框里，便于继续编辑或直接发送

## 当前最关键的已知风险

### 风险 1：误把试探句识别成任务

说明：

- 这类问题之前已经出现过
- 典型表现是 `测试一下这个系统` 被误建成收集箱任务

当前状态：

- 已经加了本地拦截
- 仍需真机回归确认是否还有边界误判

### 风险 2：明确建任务语句被误伤

说明：

- `帮我记一个任务` 这类明确录入句，之前曾被过度拦截

当前状态：

- 已修正优先级
- 仍需回归确认不要被测试句逻辑再次误伤

### 风险 3：结果卡太像系统日志

说明：

- 真正执行任务时，结果卡保留是对的
- 但文案如果太技术化，会让整个产品像后台工具

当前状态：

- `capture_message` 和 `summarize_lists` 已不再出卡
- 仍需继续检查 `create_reminder / move / complete` 的卡片气质

### 风险 4：查看类回复不够自然

说明：

- 目前已经从“列表数量”进步到“任务汇报”
- 但还需要继续贴近飞书版 AIGTD 的秘书感

当前状态：

- 已支持 `今天 / 明天 / 当前 / 某清单`
- 仍需检查是否存在“半结构化、像查询结果”的口吻

### 风险 5：远端模型与本地 fallback 口吻不一致

说明：

- 本地规则已经更像 AIGTD
- 远端模型可能仍带结构化倾向

当前状态：

- 已做 prompt 和文档注入
- 仍需检查真实联网模型时的语气是否稳定

### 风险 6：语音输入尚未接入真流式回填

说明：

- 当前对话输入仍是键盘为主
- 如果后续接豆包语音识别，接口、回填、权限、错误态都要重新验收

当前状态：

- 已有方向和初步代码
- 还需要补成更像 IM 的实时回填方式

### 风险 7：输入区联动不足

说明：

- 键盘弹出后，历史消息可能被输入框遮挡
- 输入框第一次点击可能不聚焦
- 发送状态可能让按钮长时间灰住，导致“卡了一下”的感知

当前状态：

- 已开始按输入区重构 PRD 处理
- 需要重点真机回归

### 风险 8：语音 partial / final 回填不顺

说明：

- 语音输入如果还是“录完再整段出字”，会明显不像豆包或飞书 IM
- partial 与 final 的切换必须清晰

当前状态：

- 已有豆包语音识别接入基础
- 需要补成更像 IM 的实时回填方式

## 建议的真机测试顺序

1. 先测试探句，确认不会误建任务。
2. 再测明确建任务语句，确认会真实写入 `Reminders`。
3. 再测查看类语句，确认不出结果卡且回复自然。
4. 再测键盘弹出与消息上移，确认输入区不遮挡历史内容。
5. 再测输入框第一次点击，确认一次就能聚焦。
6. 再测发送按钮状态，确认点击后不出现明显卡顿。
7. 再测语音输入，确认 partial / final 文本都会回填。
8. 最后连续发 3 到 5 轮，观察上下文连贯性和回复气质。

## 通过标准

- 明确建任务会真实进入 `Apple Reminders`
- 试探句不会误建任务
- 查看类回复自然，不出结果卡
- 真动作有结果卡，且不暴露技术字段
- 整体对话气质接近飞书里的 AIGTD，而不是接口解析器
- 输入区与语音输入的手感接近飞书 IM / 豆包

## 后续建议

- 继续收紧成功结果卡样式
- 继续增强查看类对话
- 后续接入豆包语音识别后，再补语音回归验收清单
