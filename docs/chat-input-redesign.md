# Chat 输入交互重构方案

## 目标

把当前 iOS App 的输入体验，重构成接近飞书 IM / 豆包 App 的交互手感：

- 输入框稳定、即时响应
- 键盘弹出时，消息区自然上移，不遮挡历史内容
- 发送后立刻有反馈，不出现“卡一下才恢复”的感觉
- 语音输入支持实时转写到输入框
- 录音结束后再做一轮最终修正
- AI 回复支持流式显示，但不拖慢输入区交互

## 当前问题

### 1. 键盘遮挡消息

现状：
- 点击输入框后，键盘弹出
- 输入框和键盘会挡住底部消息
- 列表不会自然上移

根因判断：
- 当前聊天页是 `ScrollView + 底部 composer`
- composer 没有严格走 `safeAreaInset(edge: .bottom)`
- 消息区也没有根据键盘和输入区高度动态补底

### 2. 发送按钮卡顿

现状：
- 点击发送后，按钮会长时间处于灰色发送中
- 直到远端开始有响应或整段完成，状态才恢复

根因判断：
- `isSending` 绑得太粗
- 当前把“发请求”“等待首包”“流式输出中”“回复完成”都塞进一个状态
- 导致用户主观感受像“卡住”

### 3. 输入框点击不稳定

现状：
- 有时第一次点击输入框没有反应
- 第二次点击才真正聚焦

根因判断：
- 外层 `ScrollView` 和 `onTapGesture` 抢焦点
- 输入区命中区域不够明确
- 滚动层、背景层和输入层的手势优先级有冲突

### 4. 语音输入没有落到输入框

现状：
- 点录音后能进入“录音中”
- 点结束后没有看到文字实时出现
- 用户感知为“语音没生效”

根因判断：
- 当前实现是“先录音成本地文件，再整体转写”
- 不是流式识别
- 也没有把部分识别结果实时回填到输入框

### 5. 输入区整体气质不像 IM

现状：
- 更像一个表单式聊天输入栏
- 不像飞书、豆包那种“边想边说、边输边回”的自然交流入口

## 新交互目标

### 输入区目标

- 永远贴底，和系统键盘联动
- 点击即聚焦，不需要二次点击
- 输入时消息列表自动避让
- 发送时按钮仍保留“可用中的活性反馈”
- 不把“发送中”做成死灰状态

### 语音输入目标

- 点一下进入录音
- 录音时输入框实时显示流式转写内容
- 停止录音后做最终修正
- 修正完成后：
  - 若开启自动发送，直接发出
  - 否则保留在输入框等待确认

### 回复体验目标

- AI 气泡先出现占位
- 首字尽快出来
- 随 `delta` 连续增长
- 不阻塞输入区继续点击和输入

## 目标交互形态

### 1. 底部 Composer

布局建议：
- 左侧：语音按钮
- 中间：多行文本输入框
- 右侧：发送按钮 / 停止按钮

状态建议：
- 空白：语音按钮高优先，发送按钮弱化
- 有文字：发送按钮高亮
- AI 流式回复中：发送按钮切为“停止生成”
- 语音录制中：左侧按钮高亮为录音态

### 2. 键盘联动

目标行为：
- 输入框获取焦点时，composer 随键盘上移
- 消息列表底部留出足够空间
- 始终能看到最后一条消息和当前输入框

实现建议：
- 聊天页改为 `safeAreaInset(edge: .bottom)` 挂 composer
- 消息列表底部 padding 根据：
  - composer 实际高度
  - 键盘高度
  动态调整

### 3. 发送按钮

目标行为：
- 点击后立刻有回弹反馈
- 不进入“整段请求期间都灰掉”的状态
- 若已经开始流式输出，允许用户继续输入下一句

实现建议：
- 拆分状态：
  - `isSubmittingRequest`
  - `isAwaitingFirstToken`
  - `isStreamingReply`
- 发送按钮表现：
  - 提交瞬间显示轻量 loading
  - 首字到达后切回正常可交互态
- 不要用长期禁用按钮来表达网络处理中

### 4. 输入焦点

目标行为：
- 点输入框一次必定聚焦
- 外层滚动不应抢走第一次点击

实现建议：
- 减少背景层 `onTapGesture` 抢占
- 输入区外层使用更明确的命中区域
- 必要时把“点击空白收键盘”换成更温和的交互，例如只在拖拽时 dismiss

### 5. 语音输入

目标行为：
- 像豆包一样，录音开始后实时转写
- 输入框里动态出现文本
- 结束后再做最终修正

实现建议：
- 语音服务协议从“一次性转写”改成“流式事件”
- 事件模型：
  - `partial(text:)`
  - `final(text:)`
  - `completed(finalText:)`
  - `failed(message:)`
- 输入框在录音期间直接绑定 partial text

## 建议的状态机

### ComposerState

- `idle`
- `typing`
- `recording`
- `transcribingPartial`
- `transcribingFinalizing`
- `sending`
- `streamingReply`

### 约束

- `recording` 与 `typing` 互斥
- `streamingReply` 不阻塞用户再次输入
- `sending` 只覆盖“发起请求到收到首个 token”这一小段时间

## 豆包 / 飞书式体验对齐点

### 飞书 IM 侧

- 点击输入框立即聚焦
- 键盘出现后列表自然上移
- 输入区不抢历史消息可见性
- 发送反馈很轻，不会出现长时间死灰按钮

### 豆包侧

- 语音识别实时出字
- 结束后会做一轮校正
- 输入框既是“实时转写承载区”，也是“最终确认区”

## 技术重构建议

### 第一阶段：先把输入手感修顺

- 改 composer 为 `safeAreaInset`
- 修键盘避让
- 修点击焦点
- 重做发送按钮状态

### 第二阶段：把语音从 batch 改成 streaming

- 重构 `VoiceTranscriptionService`
- 支持流式 partial / final 回调
- 豆包 WebSocket 实时识别结果直接灌进输入框

### 第三阶段：把回复流式体验做完整

- 首 token 更快落到气泡
- 回复中允许继续输入
- 需要时支持“停止生成”

## 实施顺序

1. 键盘避让与 composer 布局重构
2. 输入焦点和点击命中修复
3. 发送按钮状态机重构
4. 语音流式转写协议重构
5. 豆包 ASR 实时回填输入框
6. 最终修正与自动发送策略
7. 回复流式期间的输入不中断

## 验收标准

- 点击输入框一次即可聚焦
- 键盘弹出后最后一条消息仍可见
- 点击发送后 100~300ms 内出现明确反馈
- 语音录制中输入框可实时看到文本增长
- 停止录音后 1 次最终修正落到输入框
- AI 回复流式增长，不阻塞下一轮输入
